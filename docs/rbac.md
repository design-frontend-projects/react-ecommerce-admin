# RBAC & Security Architecture

How authentication, authorization, navigation, and the access-control catalog
fit together in Bluewave POS, and how to extend them.

## Principles

1. **The server is authoritative.** Every `/api` route enforces permissions
   through `withAuth`. Client-side gating (`Can`, `CanAction`, `RequireScreen`,
   hook guards) is UX only — it hides what a user cannot do, it does not
   protect anything.
2. **Nothing is hardcoded.** Modules, screens, buttons, permissions, and role
   grants live in the database. Navigation is generated from that catalog.
3. **One authorization path.** `requireAuth` → `resolveEffectivePermissions`
   is the only server-side mechanism. Role-name checks and the legacy
   `profiles.role` column are not authorization inputs.

## Permission taxonomy

Canonical names are 3-part, lowercase snake_case segments:

```
<module>.<resource>.<action>      e.g. inventory.stock.manage
```

- **module** matches an `app_modules.code` (`general`, `inventory`,
  `restaurant`, `access_control`, `lookups`, `system`).
- **action** is normally the binary `view` / `manage`. Sanctioned exceptions:
  `create` (orders), `access` (POS), `use` (shifts), plus catalog-generated
  button actions (`approve`, `pay`, …).

Screen-button permissions are generated by the seeder as
`<module>.<screen>.<button>`.

**Legacy 2-part names** (`users.manage`, `inventory.view`, …) remain valid
during the transition: `LEGACY_PERMISSION_ALIASES` in
`src/features/users/data/rbac.ts` maps them to canonical names, and
`hasPermission` / `resolveEffectivePermissions` accept either spelling in
either direction. This keeps pre-rename sessions, cached client stores, and
any un-migrated caller working. Remove the alias map only once no stored
permission row or client bundle uses the old spelling.

Wildcards: `*` (everything), `<module>.*`, `<module>.<resource>.*`.

## Resolution order

```
user_permissions (deny)  ──▶ wins over everything, including `*`
user_permissions (grant) ──▶ added on top of roles
user_roles → roles → role_permissions
DEFAULT_ROLE_PERMISSION_NAMES wildcard (super_admin → `*`)
```

`resolveEffectivePermissions` (`src/features/users/data/rbac.ts`) implements
this and is shared verbatim by the server gate and the client bootstrap, so
the two can never diverge.

## Server request flow

```
fetch('/api/…', Bearer <supabase access token>)
  └─ withAuth(PERMISSIONS.X, handler)        src/server/utils/with-auth.ts
       ├─ getBearerToken            → 401 when missing/blank
       ├─ requireAuth(token, perms) → 401 invalid session / 403 insufficient
       │    ├─ supabaseAdmin.auth.getUser   (JWT, never cached)
       │    └─ permission resolution        (cached, AUTH_CACHE_TTL_MS)
       ├─ handler({ request, params, auth, getTenantId })
       └─ uncaught error            → 500, `{ success, error }` envelope
```

- `auth` is `{ userId, primaryRole, roleNames, permissionNames }`.
- `getTenantId()` is lazy and memoized per request.
- Requirement forms: a single permission, an array (ANY-of), or
  `{ allOf: [...] }`.

### Permission cache

`AUTH_CACHE_TTL_MS` (default `30000`, `0` disables) caches the DB permission
lookup per auth user. JWT verification is never cached. RBAC mutations call
`invalidatePermissionCache()`, so changes apply immediately on the writing
instance; other instances converge within the TTL.

## Client gating

| Component / hook | Use for |
|---|---|
| `<Can permission role>` | show/hide any element |
| `<CanAction screen button>` | action buttons mapped through `screen_buttons` |
| `<RequireScreen>` | route-level guard, mounted once in `_authenticated/route.tsx` |
| `useAuthQuery({ rbac })` | disable a query the user may not run |
| `useAuthMutation({ rbac })` | pre-flight check before a write |

Permissions reach the client via `useCurrentUserAccess` → `useRBACStore`, and
refresh live through Supabase realtime subscriptions on `user_roles`,
`role_permissions`, `tenant_users`, and `profiles`.

## Navigation

`GET /api/access-control/navigation` returns the modules/screens the caller
may see (filtered server-side) plus a `route → allowed` map covering every
active screen, and the `screen → button → permission` map used by
`CanAction`.

- Flag `VITE_DB_NAV=true` switches the sidebar to the DB catalog; otherwise
  the hardcoded `sidebar-data.ts` renders. The hardcoded list is also the
  fallback when the query errors or the catalog is empty.
- `RequireScreen` **fails open** for routes not in the catalog (so new routes
  are reachable before they are registered) and denies only routes the
  catalog explicitly marks as not allowed.

## Adding things

**A permission**
1. Add it to `BASE_PERMISSION_DEFINITIONS` and a legacy alias if it replaces
   an old name (`src/features/users/data/rbac.ts`).
2. Add the constant to `PERMISSIONS` (`permission-constants.ts`) — the parity
   test enforces the two stay in sync.
3. Add an idempotent migration inserting the row and granting it to the
   appropriate default roles.
4. Grant it in `DEFAULT_ROLE_PERMISSION_NAMES` for new tenants.

**A screen**
1. Add a `SCREEN_SEEDS` entry (code, name, route, moduleCode, roles,
   permissions) and bump `SEED_VERSION` in
   `src/features/access-control/data/seed-data.ts`.
2. Create the route file; the router regenerates `routeTree.gen.ts`.
3. Optionally add an icon key to `icon-registry.ts` — the screen code is used
   as the fallback lookup.

Existing catalogs can also be edited at runtime under
`/access-control/screens` and `/access-control/buttons`; the seeder never
overwrites admin customizations (mapping rows are only created when a parent
has none).

**A protected endpoint**

```ts
const POST = withAuth(PERMISSIONS.INVENTORY_MANAGE, async ({ request, auth }) => {
  // domain validation → 400 via jsonError, everything else → 500
  return Response.json({ success: true, data: await doWork(auth.userId) })
})
```

Never accept a caller identity from the request body — derive it from `auth`.

## Audit

Every RBAC mutation writes to `rbac_audit` (actor, tenant, action, target,
JSON diff) via `recordRbacAudit`, which is best-effort and never fails the
mutation. `role_permissions.granted_by`, `user_permissions.granted_by`, and
`user_roles.assigned_by` record the acting admin. Admins read their tenant's
trail at `/access-control/audit` (`access_control.audit.view`); system owners
see all tenants.

## Known gaps

- ~299 direct `supabase.from(...)` client reads still bypass the API layer and
  depend on RLS, which is only partially defined in-repo. Migrating these is
  tracked as the final refactor phase.
- `RequireScreen` and the DB sidebar are behind `VITE_DB_NAV` until the
  catalog is verified against every route in a real environment.
- Tenant-scoped custom roles have the `roles.tenant_id` column, but global
  name uniqueness is retained until role creation is made tenant-aware.
